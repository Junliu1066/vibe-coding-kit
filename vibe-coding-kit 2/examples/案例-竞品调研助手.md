# 案例：用 vibe-coding-kit 做一个「竞品调研助手」

> 这是一份**端到端走查**：一个不写代码的产品经理，怎么用本套件的 4 个 Skill，把"竞品调研助手"从一句话想法，一路做到能用、能（选择性）上线。
>
> 重点不在代码，而在**每一步的具体决定**——列了哪些方案、砍掉哪个、为什么。照着这个套路，你自己的项目也能这么走。

**项目一句话：** 我给工具一组竞品的链接，它抓取公开信息、用 AI 提炼成一份结构化对比报告，省下我手工翻官网做表格的几个小时。

---

## 第 1 步 · 需求对齐（vibe-coding-requirements）

### 阶段零：先别急着写代码

| 自检 | 这个项目的回答 |
|------|--------------|
| 有没有现成工具？ | 有竞品监控类 SaaS，但要么贵、要么字段不是我要的、要么不让我自定义 AI 总结的角度。 |
| 一次性还是反复做？ | **反复做**——每进一个新方向都要调研一轮，值得做成工具。 |
| 自己用还是给很多人用？ | 先**自己用**，可能以后给团队几个同事用。 |

结论：值得自己做，而且先做"自己用"的最简版。

### 阶段一：四要素 → 需求基准描述

> **场景：** 我（产品经理）一个人用，在自己电脑上跑，每次调研新方向时手动触发。
> **目标：** 我准备一份竞品清单（名字 + 官网/应用商店链接），工具抓取每个竞品的公开信息，用 AI 提炼成对比表：定价、核心功能、最近动态、用户评价要点，最后导出一份报告我能直接看。
> **约束：** 我不写代码（用 AI 写）；希望尽量简单、本地能跑；会用到 AI 接口（按量收费，得控制成本）；抓的是公开信息，不碰用户隐私数据；原始链接清单别弄丢。
> **验收标准：** 我给 5 个竞品链接 → 运行 → 拿到一份含上述 4 个维度的对比报告（Markdown 或 Excel）→ 内容大体准确、能直接用。

### 阶段一的两个关键提问（最容易漏）

- **数据要不要长期保存？** demo 阶段不用——每次出一份报告文件就行。（以后想"存历史、对比变化"再说。）
- **会不会花钱？** 会，AI 接口按量收费。先问 AI:"调研 5 个竞品大概要调用几次 AI、花多少钱？" 心里有数再开跑。

> 这就是 demo 阶段需要的全部。把上面这段发给 AI 让它开干即可。下面是"决定做成正式系统"后才需要的。

---

## 第 2 步 · 看懂架构 + 选型（vibe-coding-architecture）

### 用 6 维度照一遍这个项目

| 维度 | 这个项目的情况 |
|------|--------------|
| 1. 数据存哪 | demo 不存，直接出文件；正式版用一个**本地数据库文件**存历史报告，关机也在。 |
| 2. 在谁机器上跑 | demo 在我自己电脑双击运行；只有当"想给同事随时在线用"时，才需要放服务器。 |
| 3. 一坨还是拆开 | **一个程序搞定**，坚决不拆。拆开是给大团队准备的负债。 |
| 4. 依赖了多少外部东西 | 两个：抓网页的工具 + 一个 **AI 接口（按量收费，是最大的外部依赖和成本来源）**。越少越好。 |
| 5. 即时还是排队 | 竞品多时"抓取+总结"会慢。demo 阶段我同步等着就行；真嫌慢，再让它"后台一个个处理、完成给我报告"。**别提前上队列。** |
| 6. 多少人扛不住 | 我自己用、最多几个同事——单程序绰绰有余。**不为想象中的千人并发提前把方案搞复杂。** |

### 五步选型法（实际跑出来的结果）

**第一步 · 列菜单**（让 AI 列了 5 个）：
1. 一个本地 Python 脚本：读链接 → 抓网页文字 → 调 AI 总结 → 导出对比表。
2. 一个带网页界面的小应用：浏览器里填竞品、点按钮、看报告、存历史。
3. 用零代码自动化平台（如 n8n）+ AI 节点串起来，不写代码。
4. 直接买一个现成竞品监控 SaaS（买，不做）。
5. 做成浏览器插件，在竞品页面上一键抓取总结。

**第二步 · 曝代价**（节选）：
- 方案 1：最简单、最好维护；缺界面，要会运行脚本；几乎零额外依赖。
- 方案 2：能存历史、给同事用；但多了"网页 + 服务器"两层复杂度。
- 方案 3：不用写代码；但平台本身要学、要订阅费，逻辑一复杂就难维护，且"看不懂里面发生了什么"。
- 方案 4：最省事；但贵、字段固定、AI 总结角度不可控——正是我想自己做的原因。
- 方案 5：体验好；但插件开发 + 浏览器兼容是另一套坑，对新手偏重。

**第三步 · 约束过滤**：我不写代码、要本地能跑、先自己用、控成本 → 砍掉方案 5（太重）、方案 4（已否决）。方案 3 留作"如果完全不想碰代码"的备选。

**第四步 · AI 协作评估**：Python 脚本 → AI 一次写通概率高、代码我能大致看懂、报错能整段复制、中文资料海量。✅

**第五步 · 维护裁决**："一年后出 bug 我还能修吗？" 一个能重跑的 Python 脚本 → 能。

> **选定：** demo 用**方案 1（本地 Python 脚本）**。等真需要"存历史 / 给同事在线用"，再升级成**方案 2 的最简形态：单进程 web 应用 + 一个本地数据库文件**。

### 把 AI 当老师（现场示范）

> 我："用大白话解释，为什么你建议用'一个本地数据库文件'存历史，而不是装个正经数据库？"
> AI（要点）：因为你一个人用、数据量小，一个文件型数据库不用单独安装和维护，复制走就是备份；等哪天多人高频写入扛不住了，再换也不迟。
>
> —— 记进项目说明书的"技术词汇表"：**"文件型数据库 = 整个数据库就是一个文件，零安装，适合小项目。"** 见三次就记住了。

## 第 3 步 · 上线准备（vibe-coding-production）

> 只有当我决定"做成正式版、给同事在线用"时才走这一步。demo 自己用可跳过。

- **开发规范**：让 AI 每次写功能时给出分支名和 commit 说明；日志记录"开始调研 / 抓取每个链接结果 / AI 调用 / 报告生成"。
- **安全基线（本项目最相关的几条）**：
  - **第 5 条最关键**——AI 接口的密钥**绝不写死在代码里**，放单独配置文件，并加进 `.gitignore`。
  - **开源专属**：这个项目我打算开源，推上去之前必须确认代码和提交历史里**没有我的 AI 接口密钥**——泄露了别人会拿去花我的钱。
  - 给同事在线用时，加个简单口令鉴权，别让链接谁都能打开。
- **部署**：让 AI 给出"装环境 → 放文件 → 配密钥 → 启动 → 开机自启 → 看日志 → 更新版本"的步骤。
- **验收清单（亲手逐条跑）**：正常 5 个链接出报告 ✓；某个链接打不开时报清楚错而不是整个崩 ✓；空清单有友好提示 ✓；AI 接口超时能重试或跳过 ✓。

---

## 第 4 步 · 全程避坑（vibe-coding-survival）

### 守住范围——这些一律进"以后再说"清单
开发途中冒出来的，全部先记下、不打断主线：
- "要不要每天自动监控竞品、有更新就提醒我？"（→ 要常驻服务器 + 定时任务，复杂度跳一档，以后再说）
- "要不要也抓社交媒体粉丝数？"
- "要不要做个好看的图表 dashboard？"

**v1 死守一句话：给链接 → 出一份对比报告。** 做好了再回头看清单。

### 红线自检——这个项目踩线了吗？
- 动钱？只有我自己付 AI 接口的账单，不替别人收付款 → **不踩红线**，但要盯着成本。
- 存别人隐私？竞品公开信息不算；但"用户评价"里可能含用户名等 → 别长期存无关个人信息，总结完即弃。
- 抓取合规？尊重网站规则、控制频率，别把人家网站请求打爆。

> 结论：**这个项目基本可以放心自己干**。对比一下需求 Skill 里"卡密发货"那个例子——那个动了钱和库存，正式上线前该找人把关。学会区分这两类，就是会判断边界了。

### 另外两件事
- **保住能用的版本**：第一次能正常出报告后，立刻把整个文件夹复制成 `竞品调研助手-能用版`。之后大胆改。
- **让 AI 证明给你看**：它说"加好了导出 Excel 功能"，我就追问"我怎么自己验证？给我步骤"，然后亲手跑一遍看到 Excel 文件，才算过。
