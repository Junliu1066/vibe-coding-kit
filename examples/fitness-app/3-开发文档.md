# 开发文档（SDD）—— 燃动 FitPulse

---

## 1. 概述

### 一句话产品定位
三合一轻量健身伴侣 PWA：跑步节拍器 + 组间歇计时 + 喝水提醒。

### 对应 PRD
`1-PRD.md`

### 本期范围（In Scope）
- 跑步节拍器：BPM 60-240 可调，三种音色（滴声/鼓点/人声），Web Audio API 实现
- 组间歇计时：自动循环 + 手动触发双模式，预置 4 种训练方案，Web Speech API 语音播报
- 喝水提醒：自定义间隔（1-480 分钟），Notification + Vibration API，运动联动加频
- AI 训练计划解析：上传图片/文本 → 远端 AI API → 结构化训练方案
- 数据持久化：IndexedDB 存储用户设置
- PWA：可安装到手机桌面，Service Worker 离线缓存
- 反馈收集入口

### 明确不做（Non-Goals）
- GPS 跑步路线追踪、地图
- 社交功能（好友、排行榜、分享）
- 用户账号/登录体系
- 云端数据同步
- 支付/付费功能
- Apple Health / Google Fit 集成
- 后端服务器（纯前端 + 仅 AI 解析调远端 API）

### 项目元信息

| 项目 | 值 |
|------|-----|
| 项目名 | `fitpulse` |
| 语言 | HTML + CSS + JavaScript（ES2020+） |
| 运行时 | 浏览器（移动端优先） |
| 默认端口 | `3000`（本地开发服务器），部署平台自动分配 |
| 配置格式 | 无 `.env`，纯前端项目；AI API Key 通过部署平台环境变量注入 |
| 包管理 | 无（零依赖纯前端） |

```

│                                         │
│  ┌──────────────────────────────────┐   │

│  │  首页卡片 / 节拍器页 / 间歇页 /    │   │
│  │  喝水页 / 反馈弹窗                │   │
│  └──────────────┬───────────────────┘   │
│                 │                       │
│  ┌──────────────▼───────────────────┐   │
│  │       App Logic (app.js)          │   │
│  │  状态管理 / 计时器 / 校验         │   │
│  └──┬──────────┬──────────┬────────┘   │
│     │          │          │             │
│  ┌──▼──┐  ┌───▼───┐  ┌──▼──────┐      │
│  │Web  │  │Web    │  │Notifi-  │      │
│  │Audio│  │Speech │  │cation   │      │
│  │API  │  │API    │  │API      │      │
│  └─────┘  └───────┘  └─────────┘      │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │        Storage (IndexedDB)        │   │
│  │  用户设置 / 历史记录              │   │
│  └──────────────────────────────────┘   │
│                                         │
│  ┌──────────────────────────────────┐   │
│  │      Service Worker (sw.js)       │   │
│  │  离线缓存 / 安装提示              │   │
│  └──────────────────────────────────┘   │
└────────────────────┬────────────────────┘
                     │ HTTPS
                     ▼
          ┌──────────────────┐
          │  远端 AI API      │
          │  (训练计划解析)    │
          │  POST /v1/messages│
          └──────────────────┘
```

**数据流**：
1. 用户操作 → UI 事件 → App Logic 更新状态 → 渲染 UI
2. 节拍器：App Logic → Web Audio API → 系统音频输出
3. 间歇计时：App Logic → setInterval 倒计时 → Web Speech API 语音播报
4. 喝水提醒：App Logic → setTimeout → Notification API + Vibration API
5. AI 解析：用户上传 → App Logic → fetch() 远端 AI API → 解析 JSON → 更新 UI
6. 持久化：App Logic → IndexedDB 读写

## 3. 端到端流程的实现映射

| # | 用户流程步骤 | 由谁实现 | 说明 |
|---|-------------|----------|------|
| 1 | 打开 APP | `index.html` + Service Worker | PWA 加载，展示首页三张功能卡片 |
| 2 | 点击"跑步节拍器" | `navTo('metronome')` | 导航到节拍器页面 |
| 3 | 调整 BPM | `updateBPM()` / `setPreset()` | 滑块或预设按钮，更新 `state.metronome.bpm` |
| 4 | 选择音色 | `selectSound()` | 更新 `state.metronome.sound` |
| 5 | 点击"开始" | `toggleMetronome()` | 创建 AudioContext，按 BPM 间隔播放节奏音 |
| 6 | 听到节奏声 | `playBeat()` (Web Audio API) | OscillatorNode 合成滴声/鼓点；人声模式用 Web Speech API 播报"1,2,3,4"（无需预录音频文件） |
| 7 | 点击"停止" | `toggleMetronome()` | 停止 AudioContext，记录本次时长到 IndexedDB |
| 8 | 返回首页 → 点"组间歇计时" | `navTo('interval')` | 导航到间歇计时页面 |
| 9 | 切换模式（自动/手动） | `setMode()` | 更新 `state.interval.mode` |
| 10 | 选方案 | `selectPlan()` | 预置方案或自定义 |
| 11 | 点"开始" | `toggleInterval()` → `runCountdown()` | setInterval 每秒倒计时 |
| 12 | 到时间语音播报 | `speak()` (Web Speech API) | `SpeechSynthesisUtterance` 播报 |
| 13 | 全部完成 | `stopInterval()` | 清除计时器，Toast 提示 |
| 14 | 上传训练计划 | `simulateAIUpload()` → `fetch()` | POST 到 AI API，解析返回 JSON |
| 15 | 点"喝水提醒" | `navTo('water')` | 导航到喝水页面 |
| 16 | 设置间隔 | `updateWaterInterval()` | 更新 `state.water.interval` |
| 17 | 到时间收提醒 | `triggerWaterReminder()` | Notification API + Vibration API |
| 18 | 关闭 APP 再打开 | IndexedDB 读取 | 从 `fitpulse_settings` 恢复所有设置 |
| 19 | 点"反馈" | `openFeedback()` | 弹窗表单，提交后存储到 IndexedDB |

## 3b. 外部服务集成

### AI 训练计划解析

| 项目 | 值 |
|------|-----|
| 服务 | Anthropic Claude API（或兼容 OpenAI 格式的任意 LLM API） |
| 用途 | 解析用户上传的训练计划图片/文本，提取训练秒数、休息秒数、组数 |
| Endpoint | `POST https://api.anthropic.com/v1/messages` |
| 鉴权 | `x-api-key: $ANTHROPIC_API_KEY`（从环境变量注入，通过边缘函数代理） |
| 超时 | 30 秒 |
| 限流 | 免费额度取决于 API 提供商；建议加前端防抖（5 秒内不重复请求） |

#### 请求示例

```json
{
  "model": "claude-haiku-4-5-20251001",
  "max_tokens": 200,
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "请从以下训练计划描述中提取：训练秒数、休息秒数、组数、方案名称。以 JSON 格式返回，字段名为 training_seconds, rest_seconds, rounds, name。如果某个字段无法提取，用 null。仅返回 JSON，不要其他文字。\n\n用户描述：我想做 Tabata，练 20 秒歇 10 秒，总共 8 组"
        }
      ]
    }
  ]
}
```

#### 响应示例

```json
{
  "id": "msg_xxx",
  "content": [
    {
      "type": "text",
      "text": "{\"training_seconds\": 20, \"rest_seconds\": 10, \"rounds\": 8, \"name\": \"Tabata\"}"
    }
  ]
}
```

#### 错误处理

| 错误码 | 含义 | 处理 |
|--------|------|------|
| 401 | API Key 无效 | 提示"服务配置错误，请联系开发者" |
| 429 | 限流 | 提示"请求太频繁，请稍后重试" |
| 5xx | 服务端错误 | 提示"AI 服务暂时不可用，请手动设置或稍后重试" |
| 超时 | 30 秒无响应 | 提示"分析超时，请检查网络后重试" |
| 解析失败 | 返回非 JSON | 提示"训练计划解析失败，请手动设置或重试" |

#### 边缘函数代理（推荐）

为避免 API Key 暴露在前端，通过 Vercel Edge Function 或 Netlify Function 做代理：

```
用户浏览器 → fetch('/api/parse-plan', { body: userInput })
           → 边缘函数注入 API Key → 转发到 Anthropic API
           → 返回解析结果 → 前端更新 UI
```

## 6. 关键数据结构 / 接口约定

### 6.1 全局状态结构（`app.js`）

```javascript
const APP_STATE = {
  metronome: {
    bpm: 170,           // number, 60-240
    sound: 'beep',      // 'beep' | 'drum' | 'voice'
    running: false,     // boolean
    startTime: null,    // number | null, Date.now() 时间戳（运行时，不持久化）
    audioCtx: null      // AudioContext | null（运行时）
  },
  interval: {
    mode: 'auto',       // 'auto' | 'manual'
    plan: 'tabata',     // 'tabata' | 'hiit' | 'strength' | 'core' | 'custom'
    trainSec: 20,       // number, 1-600
    restSec: 10,        // number, 1-600
    rounds: 8,          // number, 1-99
    running: false,     // boolean
    currentRound: 0,    // number
    phase: 'rest',      // 'train' | 'rest'
    timerId: null       // number | null (setInterval ID)
  },
  water: {
    interval: 45,       // number, 1-480 (分钟)
    enabled: true,      // boolean
    sportLink: true,    // boolean
    todayCount: 0,      // number
    timerId: null,      // number | null (setTimeout ID)
    lastReminderDate: null // string | null, "YYYY-MM-DD" (用于每日重置计数)
  }
};
```

### 6.2 IndexedDB 数据库设计

**数据库名**：`fitpulse_db`
**版本**：`1`

#### 表：`settings`

IndexedDB object store 定义（NoSQL，非 SQL 表）：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `key` | string | 主键（object store 的 keyPath） | 设置键名 |
| `value` | object (JSON) | NOT NULL | 设置值（结构化对象，无需 JSON.stringify） |

存储键：
- `"metronome"` → `{ bpm: 170, sound: "beep" }`
- `"interval"` → `{ mode: "auto", plan: "tabata", trainSec: 20, restSec: 10, rounds: 8 }`
- `"water"` → `{ interval: 45, enabled: true, sportLink: true, todayCount: 0, lastReminderDate: "2026-06-23" }`

#### 表：`feedback`

IndexedDB object store 定义：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `id` | number | 自增主键（autoIncrement: true） | 自增 ID |
| `message` | string | NOT NULL | 反馈内容 |
| `contact` | string | 可为空 | 联系方式（选填） |
| `created_at` | string (ISO 8601) | NOT NULL, 创建时自动填入 `new Date().toISOString()` | 提交时间 |

**创建代码示例**：
```javascript
const request = indexedDB.open('fitpulse_db', 1);
request.onupgradeneeded = function(event) {
  const db = event.target.result;
  // settings: keyPath = 'key'
  db.createObjectStore('settings', { keyPath: 'key' });
  // feedback: autoIncrement = true
  const feedbackStore = db.createObjectStore('feedback', { autoIncrement: true });
  feedbackStore.createIndex('created_at', 'created_at', { unique: false });
};
```

### 6.3 API 接口约定

#### POST `/api/parse-plan`

- **用途**：解析用户上传的训练计划
- **由**：Vercel Edge Function 或 Netlify Function 实现
- **请求**：

```json
POST /api/parse-plan
Content-Type: application/json

{
  "input": "我想做 Tabata，练 20 秒歇 10 秒，总共 8 组",
  "type": "text"  // "text" | "image"
}
```

- **成功响应** (200)：

```json
{
  "success": true,
  "data": {
    "training_seconds": 20,
    "rest_seconds": 10,
    "rounds": 8,
    "name": "Tabata"
  }
}
```

- **部分成功响应** (200)：

```json
{
  "success": true,
  "data": {
    "training_seconds": 40,
    "rest_seconds": 20,
    "rounds": null,
    "name": null
  },
  "warning": "部分字段未识别，已用默认值填充"
}
```

- **错误响应** (4xx/5xx)：

```json
{
  "success": false,
  "error": "AI 服务暂时不可用",
  "code": "SERVICE_UNAVAILABLE",
  "retry": true
}
```

**错误码枚举**：

| code | HTTP 状态 | 含义 |
|------|-----------|------|
| `INVALID_INPUT` | 400 | 输入为空或格式不支持 |
| `RATE_LIMITED` | 429 | 请求太频繁 |
| `SERVICE_UNAVAILABLE` | 503 | AI 服务不可用 |
| `PARSE_FAILED` | 422 | AI 返回无法解析 |
| `TIMEOUT` | 504 | 分析超时 |

## 7. 技术选型（已定）

| 维度 | 选择 | 理由 |
|------|------|------|
| 语言 | HTML + CSS + JavaScript (ES2020+) | 用户选定，零依赖，最轻量 |
| 音频 | Web Audio API | 浏览器原生，无需外部库，节拍精度 ±5ms |
| 语音 | Web Speech API (SpeechSynthesis) | 浏览器原生，离线可用 |
| 通知 | Notification API + Vibration API | PWA 标准推送方式 |
| 存储 | IndexedDB | 浏览器原生，支持结构化数据，容量大 |
| 离线 | Service Worker + Cache API | PWA 标准离线方案 |
| 部署 | Vercel（免费额度） | 零配置、自动 HTTPS、边缘函数支持 |
| AI API | Anthropic Claude Haiku 4.5 | 最便宜够用，JSON 结构化输出稳定 |

**不选其他方案的原因**：
- React/Vue：用户选定纯 HTML，不需要框架复杂度
- 原生 APP：开发成本翻倍，应用商店审核门槛
- SQLite/PostgreSQL：纯前端无需服务端数据库

## 8. 跑在哪 / 部署

### 方式一：本地运行（适合开发调试）

#### 第 1 步：安装 Node.js（如果还没装）

- 打开 [nodejs.org](https://nodejs.org)
- 下载 LTS 版本（左侧绿色按钮）
- 双击安装，一路"继续"即可
- **为什么**：需要一个本地服务器来运行 PWA（Service Worker 需要 HTTPS 或 localhost）
- **验证**：打开终端，运行 `node --version`，看到版本号（如 v20.x）即可

#### 第 2 步：安装本地服务器

- 打开终端（Mac：按 `Cmd+空格` 搜索"终端"）
- 运行：`npm install -g serve`
- **为什么**：`serve` 是一个简单的本地静态服务器，能让你的项目像真实网站一样运行
- **验证**：运行 `serve --version`，看到版本号即可

#### 第 3 步：启动本地服务器

- 在终端中进入项目目录：`cd ~/Downloads/vibe-coding-kit_副本/健身APP-需求`
- 运行：`serve .`
- 终端会显示一个地址，如 `http://localhost:3000`
- **为什么**：这样 PWA 的所有功能（通知、Service Worker）才能正常工作
- **验证**：在浏览器中打开 `http://localhost:3000`，能看到首页三张功能卡片

#### 第 4 步：每次修改代码后

- 修改 HTML/CSS/JS 文件后，刷新浏览器页面即可看到变化
- 如果改了 Service Worker（`sw.js`），需要在浏览器中按 `Cmd+Shift+R` 强制刷新

### 方式二：托管平台部署（推荐，适合正式上线）

#### 第 1 步：注册 Vercel

- 打开 [vercel.com](https://vercel.com)
- 点击 "Sign Up"，用 GitHub 账号注册（没有 GitHub 先去 [github.com](https://github.com) 注册）
- **为什么**：Vercel 免费托管静态网站 + 边缘函数，自动 HTTPS
- **验证**：注册成功后能看到 Vercel 控制台 Dashboard

#### 第 2 步：安装 Vercel CLI

- 打开终端，运行：`npm install -g vercel`
- **为什么**：Vercel CLI 让你能在终端里一键部署，不用手动上传文件
- **验证**：运行 `vercel --version`，看到版本号即可

#### 第 3 步：部署

- 在终端中进入项目目录：`cd ~/Downloads/vibe-coding-kit_副本/健身APP-需求`
- 运行：`vercel`
- 第一次会提示登录，按提示在浏览器中授权
- 按提示确认项目设置：
  - "Set up and deploy?" → 输入 `Y`
  - "Which scope?" → 选择你的账号
  - "Link to existing project?" → 输入 `N`
  - "Project name?" → 输入 `fitpulse`（或直接回车用默认）
  - "In which directory is your code?" → 直接回车（当前目录）
  - "Want to override the settings?" → 输入 `N`
- **为什么**：这一步把代码上传到 Vercel 的服务器，自动分配一个公网可访问的 URL
- **验证**：完成后终端会显示一个 URL（如 `https://fitpulse.vercel.app`），在浏览器中打开能看到首页

#### 第 4 步：配置环境变量

- 在 Vercel 控制台 → 项目 Settings → Environment Variables
- 添加：`ANTHROPIC_API_KEY` = 你的 API Key
- **为什么**：AI 解析功能需要这个密钥，放在环境变量里不会暴露在前端代码中
- **验证**：在间歇计时页面点击"上传训练计划"，AI 解析功能正常工作

#### 第 5 步：设置自定义域名（可选）

- 在 Vercel 控制台 → Settings → Domains
- 添加你的域名，按提示配置 DNS
- **为什么**：让用户通过你自己的域名访问，而不是 `vercel.app` 的子域名

#### 第 6 步：每次更新代码

- 修改代码后运行：`vercel --prod`
- 或连接 GitHub 仓库，推送代码自动部署
- **为什么**：`--prod` 表示部署到生产环境，确保用户访问的是最新版本
- **验证**：打开你的 `.vercel.app` 域名，能看到更新后的内容

## 9. 安全底线

| 规则 | 实现方式 |
|------|----------|
| API Key 不入前端 | 通过 Vercel Edge Function 代理，Key 仅存在于环境变量 |
| 输入校验 | 所有用户输入在前后端各校验一次：BPM (60-240)、训练/休息秒数 (1-600)、组数 (1-99)、喝水间隔 (1-480) |
| XSS 防护 | 所有用户输入在渲染前经过 `textContent`（非 `innerHTML`）处理 |
| 错误信息不暴露内部细节 | 前端错误提示仅显示用户友好文案，不暴露堆栈 |
| HTTPS 强制 | Vercel 自动提供，HTTP 自动跳转 HTTPS |
| Content-Type 嗅探防护 | `X-Content-Type-Options: nosniff` |
| 通知权限 | 仅在用户主动操作后请求通知权限，不自动弹 |

## 13. 开发规范

本项目的研发操作系统按以下 9 项体系落地：

1. **标准研发流水线**：需求（1-PRD.md）→ 架构（本文档 §2）→ 任务拆分（本文档 §11）→ 先写测试（本文档 §12）→ 写代码 → 功能测试 → 安全测试 → 部署 → 验证。按序不跳步。

2. **角色分工**：开发 AI 在动手时显式声明当前角色。产品架构组：维护 PRD/SDD；开发组：写 HTML/CSS/JS；测试组：按 §12 执行测试；运维组：Vercel 部署；治理组：ADR 记录。

3. **分区开发**：`main` 分支保持可部署状态；`develop` 分支日常开发；`feature/xxx` 分支做新功能。绝不直接改 `main`。

4. **红绿测试**：核心功能（节拍器 BPM 精度、间歇倒计时逻辑）先写会失败的断言，再写最小代码通过。

5. **安全测试**：Prompt Injection 防护（AI 解析 prompt 中加护栏）；危险命令硬阻断（无 eval、无 innerHTML）；API Key 不入前端。

6. **压力评估**：当前为单用户本地 PWA，无并发压力。瓶颈在 AI API 调用（每分钟 5 次限流），超限时返回 429 + 提示。

7. **评测体系**：建黄金样本——节拍器 180 BPM 时 10 秒应有 30 次节拍；间歇计时 20 秒误差不超过 ±0.5 秒。改动后回归验证，掉分不发布。

8. **运维/可观测性**：前端关键操作（开始/停止节拍器、间歇开始/完成、喝水提醒触发）打 `console.log` 带时间戳；Vercel 边缘函数日志记录请求/响应/耗时。

9. **知识沉淀/ADR/治理**：关键决策（为什么选纯 HTML 而不是 React、为什么选 Web Audio API 而不是预录音频文件）记录为 ADR，存放在 `docs/adr/` 目录。

## 15. 反馈回路

### 15.1 铁律

**开发 AI 在以下情况必须暂停，不得自行决定修改方向：**
- 发现某个需求在技术上无法实现
- 发现两个需求之间存在矛盾（做了 A 就做不了 B）
- 发现按照本 SDD 做出来的东西和用户预期明显不符
- 遇到本 SDD 没有覆盖、但必须做决策的岔路

### 15.2 暂停后的行动步骤

1. 开发 AI 用大白话向用户解释："遇到了什么问题、为什么卡住了、有几个选项"
2. 建议用户**带着具体问题回到需求文档的作者那里**确认修改方向
3. 拿到更新后的需求文档 / SDD 再继续开发
4. 如果用户说"你先按你的判断做"，开发 AI 仍需先记录"此处偏离了原 SDD，因为______"，然后才动手

### 15.3 修改记录的格式

如果需求文档因此被修改，每次修改在 `revision_log` 中追加一条：

```
- 版本 N：______（原因）
  - 改了：______
  - 改了：______
```
