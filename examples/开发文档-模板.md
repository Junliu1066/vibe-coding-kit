## 13. 开发规范

<!-- 来自 vibe-coding-production §开发规范 + survival 的纪律。让开发 AI 遵守。按需精简。 -->
1. **研发流水线**：需求(1-PRD.md) → 架构(§2) → 任务拆分(§11) → 先定测试(§12) → 写代码 → 功能测试 → 安全测试 → 部署 → 验证。按序不跳步。
2. **分支策略**：`main` 永远可部署；`develop` 日常开发；`feature/xxx` 做新功能。绝不直接改 `main`。（不用 git 则用「复制能用版文件夹」兜底。）
3. **日志**：关键操作打 `[时间] [级别] [模块] 内容`，敏感信息脱敏。
4. **错误处理**：所有可能出错处显式处理，错误信息含「哪里错/为什么/怎么办」，绝不静默吞错。
5. **代码风格**：注释中文；每个函数写「做什么/输入/输出」；变量函数名英文且见名知意。
6. **安全**：无 `eval`、无 `innerHTML` 拼接；密钥不入前端；输入校验。
7. **知识沉淀**：关键技术决策记一句「为什么这么选」，便于日后回看。

## 14. 给开发 AI 的话

- **严格按本 SDD 和 PRD 实现**；所有 UI 照 `2-交互图.html` 还原，色值从 PRD 的设计 Token 提取。
- **按里程碑顺序来**，先 M0 核心跑通，不要一次堆完所有功能。
- **每完成一个 Task，对照 §11 验收标准自检**，并告诉用户「怎么自己验证」（打开哪里 → 点什么 → 看到什么）。
- **用户不会写代码**：任何要紧的偏离，先用大白话解释、得到同意再做。
- **遇到矛盾 / 做不出来 / 和预期不符 / 文档没覆盖的岔路 → 立刻暂停**，向用户说清「什么问题、有哪些选项」，建议回去改需求文档，拿到更新再继续。绝不自行决定方向。
- **用户开发中提新想法**：先记进 PRD 的「以后再说」，本期不实现，告诉用户「等这版跑通验证后再加」。
