# 阶段三：文档生成（生成三件套 + 开发预估）

## 前置条件

运行本子 skill 之前，请确认已从阶段二（`prd-solution-design`）获取了完整的状态表 JSON，其中至少包含：

- `decisions.scenario`、`goal`、`competition_check`、`validation`、`constraints`
- `decisions.end_to_end_flow`（端到端流程）
- `decisions.processing_rules`（处理规则）
- `decisions.tech_solution`（技术方案）
- `decisions.gaps_filled`（补全的默认项）
- `phase1_done = true`、`phase2_done = true`

**如果状态表不完整，不要凭空猜测。告诉用户"还有些信息需要补全，请回到前面的阶段"。**

## 生成三件套

对齐一致后，产出**三个文件**，存到当前目录下一个新文件夹里（如 `我的项目-需求/`）：

| 文件 | 管哪一层 | 专业级标准 |
|------|---------|------|
| `1-PRD.md` | 产品层：做什么、规则是什么 | 一份**专业产品经理水准**的 PRD |
| `2-交互图.html` | 画面层：长什么样、怎么点 | 一个**设计精良、可交互**的高保真原型 |
| `3-开发文档.md` | 技术层：架构/模块/接口/任务/验收 | 一份**架构师水准、能支撑生产级开发**的 SDD |

**所有内容来自状态表 `decisions`，不要凭空加需求。** 在专业度和完整度上，按交付正式商业项目的标准来要求自己。

---

### `1-PRD.md`（产品需求文档）

按专业产品经理的 PRD 标准写。所有内容从 `decisions` 读取。至少包含：

```markdown
# 产品需求文档（PRD）

## 一句话
<!-- 从 decisions.goal.problem 提炼 -->

## 背景与问题
<!-- decisions.goal.problem -->

## 目标用户与场景
<!-- decisions.scenario.who、decisions.scenario.device -->

## 产品目标与成功指标
- 目标：______
- 衡量成功的指标：______（呼应 decisions.validation）

## 核心功能（必须做到，按优先级）
1. [P0] ______
2. [P0] ______
3. [P1] ______
<!-- 从 decisions.end_to_end_flow 提炼 -->

## 核心功能的处理规则（开发必须严格照做）
<!-- 从 decisions.processing_rules 逐条列出 -->
- 清洗/规范化：______
- 重复怎么办：______
- 缺失/不完整怎么办：______
- 怎么算一条有效记录：______
- 排序/分组：______
（标「默认，可改」的是 decisions.gaps_filled 中的项）

## 出错和边界
- ______（空输入、识别失败、网络断等，分别给什么提示）

## 限制
- 给谁用 / 长期还是一次性：______（decisions.constraints.user_count）
- 数据存不存：______（decisions.constraints.data_persistence）
- 花钱：______（decisions.constraints.has_cost）

## 怎么算做好了（验收）
我做 ______，应该看到 ______（decisions.acceptance）

## 验证目标（这 demo 拿去市场要验证什么）
- 最想验证：______（decisions.validation.goal）
- 算验证成功的信号：______（decisions.validation.success_signal）
- 为此 demo 里要带的小入口：______（decisions.validation.entry_in_demo）

## 验证通过后的商用待办（仅商用项目）
<!-- 如果 decisions.scenario.is_commercial = true，读 references/商用补充.md 写入此节 -->
```

### `2-交互图.html`（能点的交互原型）

生成**一个独立的、能直接双击打开的 HTML 文件**——**设计精良的高保真可交互原型**。

要求：

- 纯 HTML + 内联 CSS（必要时少量 JS），**不依赖任何外部文件/网络**，双击即用
- **视觉要专业**：合理的配色、间距、字体层级、圆角阴影、按钮状态
- 把 PRD「核心功能」里的每个界面元素都摆上，布局合理
- **响应式**：手机和电脑都好看
- **真实可交互**：关键操作用预置假数据演示真实效果（点按钮→表格出数据、可编辑、状态变化）
- 顶部注明"高保真交互原型（演示数据为示例）"

生成后**先自检一遍**：PRD「核心功能」里提到的每个界面元素和交互，原型里是不是都有了。缺的补上。

### `3-开发文档.md`（能真正支撑开发的 SDD —— 别写薄了）

这是三件套里最决定"能不能真开发出来"的一份。它不是"交代几句技术方向"，而是一份**给"用户借手开发的那个 AI"看的施工图（SDD）**。

**严格照 `references/开发文档结构.md` 来产出**（在 prd-helper 的 references 目录）。必须包含：

| 章节 | 要求 |
|------|------|
| 1. 概述 | 一句话产品定位、本期范围、non-goals |
| 2. 总体架构 | 文本框图：主要组成部分和数据流动 |
| 3. 端到端流程的实现映射 | 用户每一步 → 哪个模块/函数/接口完成（从 decisions.end_to_end_flow 映射） |
| 4. 模块清单 | 模块名 \| 文件路径 \| 职责 |
| 5. 目录结构 | 完整的文件树 |
| 6. 关键数据结构 / 接口约定 | 数据结构字段、API 输入输出 |
| 7. 技术选型（已定） | decisions.tech_solution 的全部内容 |
| 8. 跑在哪 / 部署 | 读 references/部署清单.md，填对应部署方式的步骤 |
| 9. 安全底线 | 密钥不入前端、输入校验等 |
| 10. 分阶段里程碑 | 先最小可用再加（M0 核心能跑通 → M1 → M2） |
| 11. 开发任务总表 | Task ID \| 阶段 \| 任务 \| 验收标准 |
| 12. 测试计划 | 功能/边界/异常测试用例 |
| 13. 开发规范 | 读 references/开发规范.md，按八大体系落地为本项目规矩 |
| 14. 给开发 AI 的话 | 严格按本 SDD 和 PRD 实现，按里程碑顺序来 |

其中"模块清单、目录结构、关键接口、任务表+验收"这几样**再小的 demo 也不能省**。

> 一句话判断 SDD 够不够格：**把它单独丢给一个没参与过对话的 AI，它能不能照着把东西建出来、且建出来的正是用户要的？** 能，才算合格。

## 开发预估（P2 新增） ★

### 为什么要有这一节

用户拿到三件套后，需要有**现实预期**——大概要花多少钱、多少时间、自己需要搞定什么。否则 ta 拿着文档去找 AI 开发，可能会被隐藏成本或技术门槛吓到。

### 生成完三件套之后，加一段开发预估

基于状态表中的 `decisions`，估算以下内容并写入对话（**不是写入 PRD/SDD，是直接对用户说**）：

> "在交给你之前，我先帮你估算一下接下来的开发门槛，让你心里有数 👇
>
> **⏱ 开发时间预估**
> 根据你的需求复杂度，我估计：
> - 如果找一个有经验的开发者：约 \_\_\_ 小时
> - 如果你是零基础、跟 AI 边学边做：约 \_\_\_ 天（每天 2-3 小时）
>
> **💰 每月运行成本预估**
> - AI 接口调用费：约 ¥\_\_\_/月（按你预估的使用量）
> - 部署平台费：约 ¥\_\_\_/月（或免费额度够用）
> - 总计：约 ¥\_\_\_/月
>
> **🔧 你需要自己搞定的几件事**
> 1. 申请一个 \_\_\_ 的 API 密钥（我教你怎么申请）
> 2. 注册 \_\_\_ 平台账号（我教你注册）
> 3. 学会双击打开 HTML 文件 / 在浏览器里打开网址
>
> **⚠️ 注意事项**
> - 如果涉及真实收付款、存别人隐私、或出错有真实后果，强烈建议找个懂行的朋友帮你把关
> - 开发中遇到技术问题，把完整报错复制给 AI 让它帮你解决
>
> 这些你先了解，不用怕——每一步我都会在开发文档里写清楚怎么带你做。"

### 开发预估的估算方法

| 因素 | 估算依据 |
|------|---------|
| 开发时间 | 端到端流程 step 数 × 15-30 分钟/步（零基础找 AI 做） |
| AI 接口费 | 每次调用费用 × 日均使用次数 × 30 天 |
| 部署平台费 | 根据选定的部署方式：本地运行 = ¥0，托管平台 ≈ ¥0-50/月 |
| 用户需自己做的事 | 根据 decisions.constraints.user_can_code 调整说明的详略 |

如果状态表中 `constraints.has_cost = false` 且技术方案不涉及付费 API，则运行成本写"约 ¥0/月"。

## 收尾的话

三个文件生成 + 开发预估说完后：

> "✅ 全搞定！我在 `我的项目-需求/` 文件夹里给你生成了三样东西：
> 1. **PRD** —— 你的需求，理得清清楚楚
> 2. **交互图（网页）** —— 双击打开就能看到东西大概长啥样
> 3. **开发文档** —— 给 AI 看的施工说明
>
> 你先打开那个交互图看看样子；觉得对了，就把这三个文件一起发给 Claude Code / Cursor，让它照着开发。哪里想改随时回来找我～"

**提醒**：如果项目涉及真实收付款、存别人的身份证/手机号等隐私、或出错有真实后果，在开发文档里明确建议用户找个懂行的人帮忙把关。

## 阶段三完成

更新状态表：`phase3_done = true`。整个流程结束。
