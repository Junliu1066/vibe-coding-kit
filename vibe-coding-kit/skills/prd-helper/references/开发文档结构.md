# 开发文档（SDD）结构蓝图 —— 生成 `3-开发文档.md` 时严格照此产出

## 为什么要这么细

`3-开发文档.md` 不是"交代几句"，它是**给"用户借手开发的那个 AI"看的施工图（SDD）**。它必须具体到：AI 照着它，能一个模块一个模块地建、一个任务一个任务地验收，**全程不用瞎猜、不用反复回头问用户技术问题**（因为用户是小白，答不了）。一份薄文档＝开发时处处卡壳。所以下面每一节都要实打实写出来，宁详勿略。

注意：三件套都是**专业级产出**，不是给小白看的简化版——小白是通过前面的对话参与决策的，不靠读懂 SDD。这份 SDD 技术密度高、读者主要是开发 AI，但**质量要按架构师交付生产级项目的标准**来要求。所有"产品层"的决定（要做什么、规则、技术形态/栈/部署）必须和前面跟用户对齐的结论一致，不能在这里偷偷改。

## 必须包含的章节（按项目裁剪，但别省核心节）

### 1. 概述
- 一句话产品定位（与 PRD 一致）。
- 对应的 PRD 文件名。
- 本期范围 / 明确不做什么（non-goals）——防止 AI 自行扩张。
- **项目元信息**（★ 必写）：
  - 项目名 / 模块名（如 `github.com/user/wechat-recon`）
  - 语言版本（如 Go ≥ 1.21、Python ≥ 3.11）
  - 默认端口（如 `8080`），以及端口是否从环境变量读取（`PORT`）
  - 配置文件格式（如 `.env`、`config.yaml`）、环境变量清单

### 2. 总体架构
用文本框图画出主要组成部分和数据怎么流动。例如一个网页工具：
```
浏览器页面(前端)
  ↓ 用户粘贴/操作
前端逻辑(收集输入、展示结果)
  ↓ 调用
小后台(serverless 函数：拿密钥、转发请求)
  ↓
外部 AI 接口
  ↓ 返回
前端渲染表格 → 导出
```
说清：哪些在前端、哪些在后台、和哪些外部服务交互。

### 3. 端到端流程的实现映射（关键，承接"拆解需求"）
把前面"拆解需求"拆出的每一步，逐条对应到"由哪个模块/函数/接口完成"。这一节保证**拆解出的核心能力真的有落点**，不会漏。例如：
| 用户流程步骤 | 由谁实现 | 说明 |
|---|---|---|
| ① 粘贴评论 | 前端输入框组件 | 校验非空 |
| ② 生成回复 | 后台 /api/generate | 调 AI，按处理规则加工 |
| ③ 手动改 | 前端可编辑表格 | … |
| ④ 复制/导出 | 前端导出模块 | … |

> **★ 关键提醒**：如果产品涉及对原始数据的**解析/提取/清洗/转换**（如 OCR 文本→结构化记录、日志解析、格式转换、AI 返回结果的后处理），这是 SDD 里最容易写薄、也最致命的一节。**不能只写"由 XX 模块处理"就完事**——必须在第 6 节（数据结构）后面或第 12 节（测试计划）中，补充以下内容：
> 1. **输入格式示例**：原始数据长什么样（至少 1 个完整示例文本）
> 2. **解析策略**：用什么方法（正则/状态机/LLM？），给出核心步骤或伪代码
> 3. **字段映射规则**：每个目标字段如何从原始输入提取（正则表达式/关键词）
> 4. **边界处理**：提取不到时的兜底策略
> 5. **输出示例**：解析完成后的结构化数据

### 3b. 外部服务集成（★ 凡调第三方 API 必写，不可跳过）

如果产品调用了任何第三方 API（AI 接口、OCR、支付、短信、地图等），必须在此节写明**精确到可直接编码**的集成细节：

- **服务名称和用途**（如"百度云 OCR 通用文字识别"）
- **调用方式**：SDK 还是裸 HTTP？如果用 SDK，写明 SDK 名和版本；如果用 HTTP，写明：
  - 完整 endpoint URL
  - 鉴权方式（API Key / access_token / OAuth 等），**给出获取 access_token 的完整流程**
  - 请求格式和响应格式的 JSON 示例
- **限流/配额**：该 API 的 QPS 限制、免费额度、超时设置
- **错误处理**：API 返回错误码时的降级/重试策略

> 不够：`调百度云 OCR 识别截图文字`
> 够：`HTTP POST https://aip.baidubce.com/rest/2.0/ocr/v1/general_basic?access_token=xxx，请求体 {image: base64编码图片}，响应 {words_result: [{words: "..."}]}，鉴权方式为 API Key + Secret Key → 调 token 接口获取 access_token（有效期30天），见伪代码`

### 4. 模块清单（带文件路径与职责）
列一张表：模块名 | 文件路径 | 职责（一句话）。文件路径要具体到能照着建。例如：
| 模块 | 路径 | 职责 |
|---|---|---|
| 页面 | `web/index.html` | 主界面结构 |
| 前端逻辑 | `web/app.js` | 收集输入、调后台、渲染表格、导出 |
| 样式 | `web/style.css` | 响应式样式 |
| 后台转发 | `api/generate.js` | 拿密钥、调 AI、按规则加工返回 |
| 配置示例 | `.env.example` | 列出需要的密钥项 |

### 5. 目录结构
给一棵完整的文件树，让 AI 知道东西都放哪。

### 6. 关键数据结构 / 接口约定

**接口定义必须包含实际 JSON 示例。** 字段名描述不够——AI 需要知道请求体和响应体的真实结构。

- 每个 API 接口给出至少 1 组请求/响应 JSON 示例，包含字段名、类型、值格式
- 定义统一的错误响应格式（如 `{"error": "..."}`）
- 字段命名在全文档保持一致（驼峰或下划线选一种）

> 不够：`POST /api/upload → 输出: { records: [...] }`
> 够：`POST /api/upload` → 响应: `{ "records": [{ "id": 1, "tx_time": "2024-06-01 14:30", "party": "张三", "direction": "入金", "amount": 123.45 }], "total": 5, "duplicated": 0 }`

**数据库设计必须包含字段约束和索引。** 每张表写明：
- 字段名、类型、长度/精度、NOT NULL、默认值
- 约束（PRIMARY KEY、CHECK、UNIQUE）
- 索引（主键索引、常用查询字段索引）
- 唯一约束冲突时的处理策略（ON CONFLICT / UPSERT）

### 7. 技术选型（已定）
承接前面跟用户对齐的结论：用什么做、为什么、为什么不选别的。更细的库选择可由 AI 定，但要写出主栈。

### 7b. 基础设施配置（★ 按项目适配，但每一项都不应缺席）

以下配置项必须在 SDD 中明确指定，不能留给开发 AI 猜测：

- **前端静态文件**：前后端分离还是嵌入二进制？如果用 Go embed，写清 `//go:embed` 路径和 Gin 挂载方式
- **CORS**：本地开发是否允许跨域？生产环境是否同域？给出具体 `AllowOrigins` 配置
- **日志**：格式（`log/slog` JSON / 纯文本）、必含字段（`level`、`msg`、`time`）、日志级别
- **图片处理**：上传前是否压缩？压缩到什么尺寸/质量？用哪个库处理？
- **限流/并发**：上传接口是否需要限流？策略（token bucket / 请求排队 / sleep）？
- **健康检查**：是否需要 `/health` 端点？部署平台用它做什么？
- **请求超时**：Gin 的 ReadTimeout / WriteTimeout 设置
- **数据库迁移**：DDL 由谁执行？（★ 必须明确——推荐"**程序启动时自动执行 CREATE TABLE IF NOT EXISTS**"或"用迁移工具 xxx"，不能让用户手动跑 SQL）

### 8. 跑在哪 / 部署
读 `references/部署清单.md`，把对应那份可一步步照做的部署清单填进来。

**必须附带完整的 `.env.example` 文件内容**（不能只列文件名——要写出每个环境变量的字段名、说明、示例值、是否必填）。例如：
```
# 必填：数据库连接
DATABASE_URL=postgres://user:pass@localhost:5432/dbname?sslmode=disable

# 必填：百度云 OCR
BAIDU_OCR_API_KEY=your_api_key
BAIDU_OCR_SECRET_KEY=your_secret_key

# 可选：服务端口（默认 8080）
PORT=8080
```

如果选择托管平台部署（如 Railway），还需写明：
- 语言版本从哪读（如 `go.mod` 中的 `go 1.xx`）
- 是否需要 Procfile 或平台配置文件
- 健康检查端点（如 `GET /api/health`）
- 端口读取方式（如"监听 `PORT` 环境变量，默认 `8080`"）

### 9. 安全底线
密钥不入前端/不写死、输入校验、对外不暴露内部错误等（按项目适配）。

### 10. 分阶段里程碑（先最小可用，再加）
把开发切成几个阶段，每阶段有明确"做完的样子"。最小阶段必须是"核心能力能跑通"。例如：
| 阶段 | 目标 | 完成标准 |
|---|---|---|
| M0 最小可用 | 粘贴→生成→显示 | 能对 3 条评论生成回复并显示 |
| M1 | 可编辑 + 导出/复制 | 能改、能复制全部 |
| M2 | 处理规则 + 边界 | 去重/空行/标记/错误提示都对 |
| M3 | 验证入口 + 部署 | 留联系方式入口可用，能在线打开 |

### 11. 开发任务总表（带 Task ID 和可验收标准）
把里程碑拆成可执行任务，每个任务一行：Task ID | 阶段 | 任务 | 验收标准（具体、可检验）。例如：
| Task ID | 阶段 | 任务 | 验收 |
|---|---|---|---|
| T-M0-01 | M0 | 搭页面骨架 | 浏览器打开能看到输入框+按钮+空表格 |
| T-M0-02 | M0 | 后台转发调通 AI | 输入 1 条评论能返回 1 条回复 |
| … | | | |
验收标准要能让人"照着点一下就知道过没过"。

### 12. 测试计划
分层列出要测什么：核心功能正常流程、处理规则、异常/边界、（涉及钱/隐私时的）安全项。

**必须提供至少 1 组具体测试数据**——输入（图片/文本/请求）→ 期望输出（结构化数据/响应/行为）。仅写"上传一张图片测试"不够，要写清楚"什么图片、期望识别出什么"。

如果产品涉及核心解析/提取逻辑（如 OCR 文本→结构化记录），提供 2-3 条测试用例覆盖：正常情况、边界（缺字段）、异常（无关文本）。每个用例给出具体的输入字符串和期望的输出结构。

### 13. 开发规范
读 `references/开发规范.md`——它是一套「研发操作系统」（标准研发流水线 + 五角色分工 + 分区开发 + 红绿测试 + 安全四层 + 压力测试 + 评测三层 + 可观测/运维 + ADR/治理）。**所有项目都按这套完整体系（C 级）全量执行**，要求开发 AI 像专业团队那样交付，不降级；只按项目真实规模调每个体系的"量"，不删项。写进本节时落地为本项目的具体规矩。

### 14. 给开发 AI 的话
- 严格按本 SDD 和 PRD 处理规则实现，照交互图还原界面。
- 按里程碑顺序来，先跑通 M0 核心，别一次堆完。
- 每完成一个任务，对照其验收标准自检，并告诉用户怎么自己验证。
- 用户不会写代码：任何要紧的偏离，先用大白话跟用户解释、得到同意再做。
- **遇到矛盾先暂停**：如果发现需求有矛盾、无法实现、或需要偏离本 SDD，不要自行决定——告诉用户"遇到了什么问题、有什么选项"，建议用户回去改需求文档。拿到更新后再继续。
- **用户开发中提新想法**：先记录下来，但不要在本期实现。告诉用户"这个可以记下来，等这版跑通验证后再加"。

### 15. 反馈回路（开发中发现矛盾怎么办，必写）

**铁律**：开发 AI 在以下情况必须暂停，不得自行决定修改方向：
- 发现某个需求在技术上无法实现
- 发现两个需求之间存在矛盾
- 发现按照本 SDD 做出来的东西和用户预期明显不符
- 遇到本 SDD 没有覆盖、但必须做决策的岔路

**暂停后的行动步骤**：
1. 开发 AI 用大白话向用户解释"遇到了什么问题、为什么卡住了、有几个选项"
2. 建议用户**带着具体问题回到需求文档的作者那里**确认修改方向
3. 拿到更新后的需求文档 / SDD 再继续开发
4. 如果用户说"你先按你的判断做"，开发 AI 仍需先记录"此处偏离了原 SDD，因为______"，然后才动手

**修改记录格式**：每次需求文档因此被修改，在 revision_log 中追加一条：
```
- 版本 N：______（原因）
  - 改了：______
  - 改了：______
```

## 裁剪原则
- 小 demo：2~10 个文件那种，第 4/5/6/10/11 节可以精简，但**模块清单、目录、关键接口、任务表+验收**这几样不能省——它们正是"能不能照着开发"的关键。
- 大项目：完整写，必要时分期产出多个 SDD 文件。
