# vibe-coding-kit 治理宪法 (Governance Constitution)

> 这是 vibe-coding-kit 的最高治理文件。Claude Code 在每次对话开始时自动加载它。
> 在其他平台（ChatGPT、Codex）使用时，将本文件内容复制为 system prompt 前缀。
> 所有规则派生自 `harness.json`——那是唯一真实来源。

---

## 一、最高原则

1. **复杂度是负债，不是资产。** 每多一个组件、一个依赖，都是向未来借债。
2. **需求没说清，等于没说。** AI 会朝着模糊的方向飞速跑偏。
3. **先说问题，再说方案。** 描述方案会锁死 AI，描述问题才有更优解。
4. **没有代价的方案不存在。** 不讲代价的推荐不可信。
5. **维护成本是最终裁决。** 一年后修不修得动，比什么都重要。

---

## 二、语言与格式

- **正文**：简体中文
- **代码注释**：简体中文
- **变量名/函数名**：英文，见名知意
- **文件编码**：UTF-8
- **行尾**：LF

| 产出物 | 路径 | 模板 |

| 开发文档 | `docs/开发文档.md` | `examples/开发文档-模板.md` |
| 进度账本 | `docs/进度账本.md` | `examples/进度账本-模板.md` |

## 四、必须产出的章节（不可跳过）

### PRD（`docs/prd.md`）至少包含：
- 0. 文档元信息（文档状态、版本、产品名称三项不可为空）
- 1. 产品结论（含核心定位 + 明确不做）
- 2. 用户、场景与形态（含"它活在哪"的明确决定）
- 9. 验收标准（每条具体可验证，不能是"系统正常"这种空话）
- 10. MVP 完成定义
- 11. 以后再说

### 项目说明书（`docs/项目说明书.md`）至少包含：
- 一句话
- 需求基准描述
- 产品思考模块（预留）
- 选定技术栈
- 验收清单
- 以后再说清单
- 决策复盘 / 案例库

### 开发文档（`docs/开发文档.md`）至少包含：
- 0. 文档元信息（文档状态、版本、产品名称三项不可为空）
- 1. 概述（含本期范围 + 本期不做）
- 2. 总体架构
- 3. 端到端流程
- 4. 模块清单
- 5. 目录结构
- 6. 数据结构与存储
- 7. 接口约定
- 8. 核心算法 / 业务规则
- 12. 分阶段里程碑
- 13. 开发任务总表
- 14. 测试计划
- 16. 反馈回路

## 五、流程纪律（按门推进）★

派生自 `harness.json` 的 `workflow` 块。**这是为了解决"AI 不按步骤推进、乱序跳步"而设的硬规矩。**

1. **以账本为状态来源。** `docs/进度账本.md` 是流程的唯一状态来源。**每轮对话开场先读它**，向用户报"当前在哪个阶段、第几步"；不存在就照模板创建。
2. **按门推进。** 流程是 S1 需求 → S2 选型 → S3 上线 的有序流水线。每个阶段有出口门（`exit_gate`），每一步有退出条件。**一步的退出条件没满足，不进下一步；上一阶段出口门没过，不进下一阶段。**
3. **每过一步回写账本。** 做完一步立刻更新账本（标状态、推进当前步骤）。
4. **跳步要留痕。** 只有分诊判定为"轻"（自己用 / 一次性 / demo）时，才允许跳 `required:false` 的步骤，且必须在账本"跳步留痕"写一行理由。`required:true` 的步骤一律不准跳。
5. **轻重两种模式。** 轻模式可"几步合并、一次报门"，但**出口门无论轻重都要过**——省的是停顿次数，不是产出质量。
6. **红线优先。** 涉及"动钱 / 动别人隐私"时，survival 的红线提示优先级高于"按门推进"——先停下来提醒找真人。
7. **问题解决后沉淀案例。** 遇到需求矛盾、缺决策、实现卡住、技术方案偏离预期时，先暂停处理；本次解决后，必须把"问题是什么、有哪些选项、最终怎么决定、下次怎么复用"写入 `docs/项目说明书.md` 的「决策复盘 / 案例库」。

## 五点五、项目进度卡纪律（低上下文恢复）

派生自 `harness.json` 的 `progress_cards` 块。它解决的是开发期上下文过重、AI 每次重读全部历史的问题。

1. **默认只读总表。** 开发期恢复上下文时，默认只读 `.dsu/progress/index.md`。
2. **按需下钻模块。** 只有当前任务涉及某个 DU / 模块时，才额外读取 `.dsu/progress/modules/duXX-xxx.md`。
3. **复盘才读任务。** 只有需要复盘具体任务、确认 closeout 或追溯改动时，才读取 `.dsu/progress/tasks/任务ID.md`。
4. **总表只放索引级信息。** 总表只回答：当前阶段、当前主线、完成 DU、进行中 DU、下一步、最大风险；细节不得堆进总表。
5. **任务结束三处同步。** 每个任务完成后，必须新建或更新任务卡，并同步更新对应模块卡和 `.dsu/progress/index.md`。
6. **状态枚举固定。** 进度卡状态只能使用：未开始、脚手架、进行中、已完成、阻塞、回滚、待重构。
7. **不默认全量读取。** 不得为了恢复上下文默认读取所有 `modules/` 和 `tasks/`。

## 六、禁止事项

1. ❌ 禁止在输出中保留模板占位符（如 `{产品名称}`、`TODO`、`TBD`、`___`）——必须填写完整
2. ❌ 禁止输出英文正文（术语、代码、URL 除外）
3. ❌ 禁止在产物中包含密钥、密码、token 等敏感信息
4. ❌ 禁止将内部架构概念（如"用了几个 agent 协作"）写进用户侧定位——这是实现细节，不是产品卖点
5. ❌ 禁止验收标准写空话（如"系统正常运行"、"功能正常"）

---

## 七、跨 Skill 一致性要求

- 项目说明书中的「需求基准描述」必须与 PRD 中的「产品结论」表述一致
- PRD 中的「形态与资源约束」必须与项目说明书中的「选定技术栈」兼容
- 开发文档必须承接 PRD 和项目说明书，把需求、技术栈和约束落到模块、接口、数据结构、任务、测试和反馈回路
- 所有产出物中的项目名称必须一致
- 验收标准必须是具体的、可手动验证的（不是"系统正常运行"）；推荐 EARS 格式：当【前置条件】，在【动作】时，则【可观察的结果】
- 「产品思考模块（预留）」只作为后续对齐入口，未定稿前不替代 PRD、不作为开发出口门、不强制补齐细项
- 「决策复盘 / 案例库」只记录可复用的判断规则和案例，不写一次性聊天过程，不记录密钥、账号、token、用户隐私原文

---

## 八、模板强制使用

- 写 PRD：必须遵循 `examples/PRD-模板.md` 的结构，逐节填写
- 写项目说明书：必须遵循 `examples/项目说明书-模板.md` 的结构
- 写开发文档：必须遵循 `examples/开发文档-模板.md` 的结构，逐节填写
- 写项目进度卡总表：必须遵循 `examples/项目进度卡-总表模板.md`
- 写模块进度卡：必须遵循 `examples/项目进度卡-模块模板.md`
- 写任务 Closeout 卡：必须遵循 `examples/项目进度卡-任务模板.md`
- 模板中的 `<!-- 注释 -->` 是写作提示，填好内容后可删除
- 颗粒度量力而行：小项目填核心章节即可，不需要全部展开

## 九、交付前自检

在声称"完成"之前，必须逐条确认：

- [ ] 所有必须产出的文件已写入 `docs/`，且非空
- [ ] 所有必须章节已填写，无空值、无占位符
- [ ] 版本号已填写，格式为 `v数字.数字`（如 v0.1）
- [ ] 文档状态已标注（草稿 / 评审中 / 交付版）
- [ ] 验收标准是具体的、可验证的
- [ ] 如本轮进入正式开发，已检查或产出 `docs/开发文档.md`
- [ ] `docs/项目说明书.md` 已保留「产品思考模块（预留）」入口
- [ ] `docs/进度账本.md` 已维护：当前阶段/步骤准确，跳步均有留痕，已完成阶段的出口门确实过了
- [ ] 如项目启用 `.dsu/progress/`，已维护总表、相关模块卡和任务 Closeout 卡，且没有默认读取全部历史
- [ ] 遇到过矛盾、缺决策、做不出来或偏离方案时，已把解决案例写入「决策复盘 / 案例库」
- [ ] 无敏感信息泄露
- [ ] 正文为简体中文

全部打勾后，可调用 `vibe-coding-harness` Skill 做最终质检。
