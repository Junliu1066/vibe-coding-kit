# 阶段三：文档生成（生成三件套 + 开发预估）

## 前置条件

运行本子 skill 之前，请确认已从阶段二（`prd-solution-design`）获取了完整的状态表 JSON，其中至少包含：

- `decisions.scenario`、`goal`、`competition_check`、`validation`、`constraints`
- `decisions.end_to_end_flow`（端到端流程）
- `decisions.processing_rules`（处理规则）
- `decisions.tech_solution`（技术方案）
- `decisions.gaps_filled`（补全的默认项）
- `phase1_done = true`、`phase2_done = true`

**如果状态表不完整，不要凭空猜测。告诉用户"还有些信息需要补全，请回到前面的阶段"。**

## 生成三件套

对齐一致后，产出**三个文件**，存到当前目录下一个新文件夹里（如 `我的项目-需求/`）：

| 文件 | 管哪一层 | 专业级标准 |
|------|---------|------|
| `1-PRD.md` | 产品层：做什么、规则是什么 | 一份**专业产品经理水准**的 PRD |
| `2-交互图.html` | 画面层：长什么样、怎么点 | 一个**设计精良、可交互**的高保真原型 |
| `3-开发文档.md` | 技术层：架构/模块/接口/任务/验收 | 一份**架构师水准、能支撑生产级开发**的 SDD |

**所有内容来自状态表 `decisions`，不要凭空加需求。** 在专业度和完整度上，按交付正式商业项目的标准来要求自己。

### `1-PRD.md`（产品需求文档）

按专业产品经理的 PRD 标准写。所有内容从 `decisions` 读取。至少包含：

```markdown
# 产品需求文档（PRD）

## 一句话
<!-- 从 decisions.goal.problem 提炼 -->

## 背景与问题
<!-- decisions.goal.problem -->

## 目标用户与场景
<!-- decisions.scenario.who、decisions.scenario.device -->

## 产品目标与成功指标
- 目标：______
- 衡量成功的指标：______（呼应 decisions.validation）

## UI/交互说明（★ 必写，必须具体到前端可直接照着做）

<!-- 用文字描述主界面的布局，至少包含： -->
- **整体布局**：主界面分几个区域、上下左右排列关系（如"上方是上传区域，下方是统计卡片，再下方是Tab切换的表格"）
- **上传组件**：按钮形式还是拖拽区？支持什么格式？上传中的进度/加载态
- **列表/表格**：列名、排序方式、行操作（编辑/删除按钮位置）
- **空态**：第一次打开/没有数据时，用户看到什么
- **加载态**：上传中/识别中/导出中的 UI
- **错误态**：OCR 失败、网络断、上传格式不对时，分别给什么提示 + 下一步操作建议
- **移动端适配**：手机上的布局和电脑上有什么不同

<!-- 如果交互图已由 2-交互图.html 覆盖，本节可简写为"详见 2-交互图.html"并概述关键交互 -->

## 核心功能（必须做到，按优先级）
1. [P0] ______
2. [P0] ______
3. [P1] ______
<!-- 从 decisions.end_to_end_flow 提炼 -->

## 核心功能的处理规则（开发必须严格照做）

<!-- ★ 关键：每条规则必须具体到 AI 可以直接写出正则/条件判断的程度，不能写模糊动词。
     如果要提取"转账记录"，必须写明：
     - 关键词/正则模式（如 `向你转账|收到转账|已收款`）
     - 关联字段的提取策略（如"时间在关键词前一行，金额跟在关键词后"）
     - 排除的边界（如"XX"不算）
     给出 1-2 个从原始文本→结构化字段的完整示例。 -->
- 清洗/规范化：______（含正则/关键词示例）
- 重复怎么办：______（含去重判断的精确条件）
- 缺失/不完整怎么办：______
- 怎么算一条有效记录：______
- 排序/分组：______
（标「默认，可改」的是 decisions.gaps_filled 中的项）

## 出错和边界
<!-- 至少包含以下场景的提示文案和系统行为： -->
- 上传非图片/不允许的格式
- OCR 返回空结果（识别不出文字）
- OCR 识别出文字但未命中任何规则（有文字但无有效记录）
- 图片模糊/质量差导致字段缺失
- 网络中断/超时
- 重复上传
- 编辑时输入非法值（如金额填非数字）
- 数据库/服务端异常时的降级提示

## 限制
- 给谁用 / 长期还是一次性：______（decisions.constraints.user_count）
- 数据存不存：______（decisions.constraints.data_persistence）
- 花钱：______（decisions.constraints.has_cost）

## 怎么算做好了（验收）

## 验证目标（这 demo 拿去市场要验证什么）

- 算验证成功的信号：______（decisions.validation.success_signal）
- 为此 demo 里要带的小入口：______（decisions.validation.entry_in_demo）

## 验证通过后的商用待办（仅商用项目）

```

### `2-交互图.html`（能点的交互原型）

生成**一个独立的、能直接双击打开的 HTML 文件**——**设计精良的高保真可交互原型**。

要求：

- 纯 HTML + 内联 CSS（必要时少量 JS），**不依赖任何外部文件/网络**，双击即用
- **视觉要专业**：合理的配色、间距、字体层级、圆角阴影、按钮状态
- 把 PRD「核心功能」里的每个界面元素都摆上，布局合理
- **响应式**：手机和电脑都好看
- **真实可交互**：关键操作用预置假数据演示真实效果（点按钮→表格出数据、可编辑、状态变化）
- 顶部注明"高保真交互原型（演示数据为示例）"

生成后**先自检一遍**：PRD「核心功能」里提到的每个界面元素和交互，原型里是不是都有了。缺的补上。

### `3-开发文档.md`（能真正支撑开发的 SDD —— 别写薄了）

这是三件套里最决定"能不能真开发出来"的一份。它不是"交代几句技术方向"，而是一份**给"用户借手开发的那个 AI"看的施工图（SDD）**。

**严格照 `references/开发文档结构.md` 来产出**（在 prd-helper 的 references 目录）。必须包含：

| 章节 | 要求 |
|------|------|
| 1. 概述 | 一句话产品定位、本期范围、non-goals |
| 2. 总体架构 | 文本框图：主要组成部分和数据流动 |
| 3. 端到端流程的实现映射 | 用户每一步 → 哪个模块/函数/接口完成（从 decisions.end_to_end_flow 映射） |
| 4. 模块清单 | 模块名 \| 文件路径 \| 职责 |
| 5. 目录结构 | 完整的文件树 |
| 6. 关键数据结构 / 接口约定 | 数据结构字段、API 输入输出 |
| 7. 技术选型（已定） | decisions.tech_solution 的全部内容 |
| 8. 跑在哪 / 部署 | 读 references/部署清单.md，填对应部署方式的步骤 |
| 9. 安全底线 | 密钥不入前端、输入校验等 |
| 10. 分阶段里程碑 | 先最小可用再加（M0 核心能跑通 → M1 → M2） |
| 11. 开发任务总表 | Task ID \| 阶段 \| 任务 \| 验收标准 |
| 12. 测试计划 | 功能/边界/异常测试用例 |
| 13. 开发规范 | 读 references/开发规范.md，按八大体系落地为本项目规矩 |
| 14. 给开发 AI 的话 | 严格按本 SDD 和 PRD 实现，按里程碑顺序来 |
| 15. **反馈回路** | 开发中发现矛盾/无法实现时，**不要自行决定**——暂停，跟用户解释，建议用户回来改需求（见下文详述） |

其中"模块清单、目录结构、关键接口、任务表+验收"这几样**再小的 demo 也不能省**。

> 一句话判断 SDD 够不够格：**把它单独丢给一个没参与过对话的 AI，它能不能照着把东西建出来、且建出来的正是用户要的？** 能，才算合格。

### SDD 各节的硬性质量要求（★ 必须遵守）

以下是对 SDD 中最容易写薄、却最关键的几个节的细化要求：

#### 第 6 节「关键数据结构 / 接口约定」

- **每个 API 接口必须给出至少 1 组请求/响应 JSON 示例**，不能只写字段名。示例要包含字段名、类型、值的格式
- 必须定义**错误响应格式**（如 `{"error": "..."}` 或 `{"code": 400, "message": "..."}`）
- 字段命名统一（驼峰还是下划线），并在全文档保持一致

> 不够：`POST /api/upload → 输入: file, 输出: { records: [...] }`
> 够：`POST /api/upload` → 请求: multipart/form-data { file: 图片二进制 } → 响应: `{ "records": [{ "id": 1, "tx_time": "2024-06-01 14:30", "party": "张三", "direction": "入金", "amount": 123.45 }], "total": 5, "duplicated": 0 }`

#### 第 6 节「数据库设计」

- 每个字段必须标注：**字段名、类型、长度/精度、NOT NULL、默认值、约束（CHECK/UNIQUE）、索引**
- 必须写明**主键索引**和**常用查询字段的索引**
- 如果涉及分页/排序，需说明排序字段的索引设计
- 如果有唯一约束，需说明冲突时的处理策略（`ON CONFLICT DO NOTHING` / `UPSERT` 等）

> 不够：`amount DECIMAL NOT NULL`
> 够：`amount DECIMAL(12,2) NOT NULL DEFAULT 0 CHECK (amount >= 0)`，加索引

#### 第 8 节（或对应的「核心算法/逻辑」节）★ 最易写薄

- 如果产品涉及**对原始数据的解析/提取/转换**（如 OCR 文本→结构化记录、日志解析、格式转换），必须给出：
  1. **输入格式的示例**：原始数据长什么样（至少 1 个完整示例）
  2. **解析策略**：用什么方法（正则、逐行解析、状态机、LLM 调用？），给出核心步骤
  3. **字段映射规则**：每个目标字段如何从原始输入中提取（正则表达式/关键词/上下文位置）
  4. **边界处理**：提取不到某个字段时的兜底策略
  5. **输出示例**：解析完成后产出的结构化数据长什么样

> 不够：`parser/parser.go → 按规则从OCR文本中提取转账记录`
> 够：提供 1 段 OCR 原始文本 → 写出逐行解析策略 + 每个字段的正则 → 给出解析后的 JSON 示例

#### 第 12 节「测试计划」

- **必须包含至少 1 组具体测试数据**：输入（图片/文本/请求）→ 期望输出（结构化数据/响应/行为）
- 对核心解析逻辑，提供 2-3 条测试用例覆盖：正常、边界（缺字段）、异常（无关文本）

#### 第 14 节「给开发 AI 的话」

- 必须包含：按里程碑顺序开发、先跑通 M0 核心再往后加、遇到矛盾先暂停、用户提新想法先记下不实现

### `3-开发文档.md` 第 15 节：反馈回路（必写内容）

生成 SDD 时，必须将以下内容写入 `3-开发文档.md` 的第 15 节「反馈回路」：

```markdown
## 15. 反馈回路（开发中发现矛盾怎么办）

### 15.1 铁律

**开发 AI 在以下情况必须暂停，不得自行决定修改方向：**
- 发现某个需求在技术上无法实现
- 发现两个需求之间存在矛盾（做了 A 就做不了 B）
- 发现按照本 SDD 做出来的东西和用户预期明显不符
- 遇到本 SDD 没有覆盖、但必须做决策的岔路

### 15.2 暂停后的行动步骤

1. 开发 AI 用大白话向用户解释："遇到了什么问题、为什么卡住了、有几个选项"
2. 建议用户**带着具体问题回到需求文档的作者那里**确认修改方向
3. 拿到更新后的需求文档 / SDD 再继续开发
4. 如果用户说"你先按你的判断做"，开发 AI 仍需先记录"此处偏离了原 SDD，因为______"，然后才动手

### 15.3 修改记录的格式

如果需求文档因此被修改，每次修改在 revision_log 中追加一条：

```
- 版本 N：______（原因）
  - 改了：______
  - 改了：______
```
```

### 给开发 AI 的话 —— 增加反馈回路约束

在 SDD 第 14 节「给开发 AI 的话」中，除了已有内容外，追加：

```markdown
- **遇到矛盾先暂停**：如果发现需求有矛盾、无法实现、或需要偏离本 SDD，不要自行决定——用大白话告诉用户"遇到了什么问题、有什么选项"，建议用户回去改需求文档。得到更新后再继续。
- **如用户在开发中提出新想法**：先记录下来，但不要在本期实现。告诉用户"这个可以记下来，等这版跑通验证后再加"。
```

## 开发预估（P2 新增） ★

### 为什么要有这一节

用户拿到三件套后，需要有**现实预期**——大概要花多少钱、多少时间、自己需要搞定什么。否则 ta 拿着文档去找 AI 开发，可能会被隐藏成本或技术门槛吓到。

### 生成完三件套之后，加一段开发预估

基于状态表中的 `decisions`，估算以下内容并写入对话（**不是写入 PRD/SDD，是直接对用户说**）：

> "在交给你之前，我先帮你估算一下接下来的开发门槛，让你心里有数 👇
>
> **⏱ 开发时间预估**
> 根据你的需求复杂度，我估计：
> - 如果找一个有经验的开发者：约 \_\_\_ 小时
> - 如果你是零基础、跟 AI 边学边做：约 \_\_\_ 天（每天 2-3 小时）
>
> **💰 每月运行成本预估**
> - AI 接口调用费：约 ¥\_\_\_/月（按你预估的使用量）
> - 部署平台费：约 ¥\_\_\_/月（或免费额度够用）
> - 总计：约 ¥\_\_\_/月
>
> **🔧 你需要自己搞定的几件事**
> 1. 申请一个 \_\_\_ 的 API 密钥（我教你怎么申请）
> 2. 注册 \_\_\_ 平台账号（我教你注册）
> 3. 学会双击打开 HTML 文件 / 在浏览器里打开网址
>
> **⚠️ 注意事项**
> - 如果涉及真实收付款、存别人隐私、或出错有真实后果，强烈建议找个懂行的朋友帮你把关
> - 开发中遇到技术问题，把完整报错复制给 AI 让它帮你解决
>
> 这些你先了解，不用怕——每一步我都会在开发文档里写清楚怎么带你做。"

### 开发预估的估算方法

| 因素 | 估算依据 |
|------|---------|
| 开发时间 | 端到端流程 step 数 × 15-30 分钟/步（零基础找 AI 做） |
| AI 接口费 | 每次调用费用 × 日均使用次数 × 30 天 |
| 部署平台费 | 根据选定的部署方式：本地运行 = ¥0，托管平台 ≈ ¥0-50/月 |
| 用户需自己做的事 | 根据 decisions.constraints.user_can_code 调整说明的详略 |

如果状态表中 `constraints.has_cost = false` 且技术方案不涉及付费 API，则运行成本写"约 ¥0/月"。
