# 阶段二：方案决策（拆解需求 → 抠规则 → 定技术方案 → 补全 → 对齐）

## 前置条件

运行本子 skill 之前，请确认已从阶段一（`prd-need-gathering`）获取了完整的状态表 JSON，其中包含 `decisions.scenario`、`goal`、`competition_check`、`validation`、`constraints`、`acceptance`。

**如果状态表缺失，先告诉用户"我需要前面理清的需求信息才能继续"，请用户提供或回到阶段一。**

## 阶段二总览

本阶段做五件事，**按顺序进行**，每一步更新状态表：

1. **拆解需求** → 还原真实端到端流程（防止"自动"做成"手动"）
2. **抠死处理规则** → 把模糊动词抠成 AI 能执行的明确规则
3. **定技术方案** → 形态 → 部署 → 技术栈，每个选项讲清代价
4. **替小白补全** → 把 ta 想不到但开发必需的空缺填满
5. **对齐确认** → 复述全貌，反复确认到 ta 说"对"

## 一、拆解需求 · 还原真实端到端流程（动手前必做）★

**核心问题：用户说的"自动"，是真的自动，还是需要用户手动做一步？**

只问"要什么功能"远远不够——拦在开发路上的坑，藏在**"然后呢？再然后呢？"** 的追问里。拿"自动识别粘贴内容并导出"举例，真正要拆出来的步骤是：

> 打开页面 → 看到输入框和示例 → 粘贴内容 → 点"识别"按钮 → 等待 → 表格出现识别结果 → 检查正误 → 手动改错 → 删除不想要的 → 点"导出" → 拿到文件

**每步都问清**：
- 这一步是**用户做**还是**系统自动做**？如果用户说是"自动"，追问"是完全不要人管、还是要人点一下才触发？"
- 这一步**谁能看见**（只有用户自己 / 管理员 / 所有人）？
- 这一步**做错了**会发生什么（给出什么提示、能不能撤回）？

把拆解结果写入 `decisions.end_to_end_flow`，格式为一个数组：

```json
"end_to_end_flow": [
  { "step": 1, "actor": "user", "action": "打开页面", "note": null },
  { "step": 2, "actor": "system", "action": "展示输入框和示例数据", "note": "第一次打开时" },
  { "step": 3, "actor": "user", "action": "粘贴内容到输入框", "note": null },
  ...
]
```

## 二、关键步骤 · 抠死「核心功能的处理规则」（开发的真正依据）

这是**最容易写模糊、也最决定开发能不能一次做对**的一步。用户说"把内容整理好"——什么叫"整理好"？你要替 ta 把每个模糊动词抠成明确规则。

### 必抠的规则清单（每个都要问清，ta 答不上来就列选项让 ta 挑）

1. **清洗/规范化**：贴进来的内容太乱怎么办？——要去掉空行、去掉特殊符号、统一格式吗？
2. **重复怎么办**：两条内容一模一样，是都留着、只留一条、还是标出来让用户选？
3. **缺失/不完整**：某条内容缺少关键信息（如没填姓名），是跳过、标红、还是让用户补？
4. **怎么算一条有效记录**：什么情况下算"一条有用数据"？
5. **排序/分组**：结果按什么顺序展示？按时间 / 按类别 / 按用户指定的方式？
6. **可不可以改**：识别/处理出来的结果，用户能不能在导出前手动修改？默认"能改"。

把规则写入 `decisions.processing_rules`：

```json
"processing_rules": [
  { "rule": "去重", "value": "完全相同的只保留第一条", "is_default": false },
  { "rule": "缺失处理", "value": "缺姓名的标红但保留", "is_default": true },
  ...
]
```

标 `is_default: true` 的表示"ta 没明确说，我替 ta 补的默认"，在 PRD 中标注「默认，可改」。

## 三、技术方案 · 替 ta 定（形态 → 部署 → 技术栈）

### 3.1 定形态（做成什么样子）

用选择题，讲代价：

> "这个东西做成什么样子？
> A. **网页（Web 应用）**：手机电脑都能打开，给个链接就行  👈 推荐，最灵活
> B. **手机 App**：像微信一样有图标，但安卓苹果得做两套，对小项目太重
> C. **浏览器插件**：在浏览器里点一下就用，但只能电脑上用
> D. **公众号/小程序**：微信里用，但要认证+审核
> E. **都不是，我说**
> 选哪个？"

更新 `decisions.tech_solution.form`。

### 3.2 定部署（跑在哪）

接上面，继续问：

> "这个东西跑在哪？
> A. **放网上（托管平台）**：手机电脑都能随时随地访问，我推荐，最省事，有免费额度
> B. **自己电脑本地跑**：最省钱但你电脑得开着才能用
> C. **听不懂，你帮我定**（那就按 A）"

**读 `references/部署清单.md`**（在 prd-helper 的 references 目录），按选定的部署方式把对应那份可一步步照做的部署清单记下来，后续写入 SDD。

涉及"动到钱/存别人隐私"的事，开发文档里要明确写"每一步部署都要带用户手把手做、讲清楚、可回退"。

更新 `decisions.tech_solution.deployment`。

### 3.3 定技术栈（别留给开发时再对）

只定"做成网页"还不够——**用什么来做，必须在这一步定下来**。小白没能力在开发时跟 AI 讨论技术栈。

技术栈 ta 完全没法自己评判，所以**不要硬塞成选择题**，而是**直接选定一套具体、主流、稳妥的技术栈**，然后**给推荐 + 讲清"为什么是它、代价是什么、别的方案差在哪"**：

> "技术上用什么，我推荐 **用最主流的网页技术（HTML/CSS/JavaScript）来做，识别那步调现成的 AI 接口。**
> · **为什么是它**：网上中文教程最多，AI 写起来一次成功率最高
> · **代价/取舍**：朴素、不追新——对你这种要稳的情况，这恰恰是优点
> · **别的方案为啥不选**：更新潮的框架资料少、AI 容易写错"

技术栈定到"语言 + 主要框架/方式 + 识别用什么"这个粒度就够了。更细的库选择留给开发 AI。

更新 `decisions.tech_solution.stack` 和 `decisions.tech_solution.why_not_others`。

## 四、核心机制 · 替小白「补全」

小白一定说不清所有细节。**你的真正价值是把 ta 想不到、却是开发必需的空缺，主动用合理默认填满。**

在确定方案之后，拿"开发必须齐的清单"自检一遍，凡是 ta 没说清或答了等于没答的地方，**替 ta 补一个合理默认，并明确标注 `（我替你先定的默认，你可改）`**：

### 开发必须齐的清单

- **核心功能的每一步**：用户具体怎么操作、点了之后发生什么。不能只写"自动识别"，要写到"粘贴 → 点按钮 → 表格里出现结果"
- **数据长什么样**：要展示哪些字段、表格有哪些列。ta 答不上来就给默认（如"预置几个常见列 + 允许用户自己加列删列"）
- **能不能改**：AI 处理难免出错，用户能不能在导出前手动修改？默认"能改"
- **出错怎么办**：处理失败、粘贴了空内容、网络断了——分别给用户看什么提示。默认"给一句中文提示，别让页面卡死或报英文错误"
- **界面大致长啥样**：一句话说清主界面有啥（如"一个大输入框 + 一个按钮 + 下面一张表格 + 一个导出按钮"）
- **空的时候/第一次打开**：用户第一次进来看到什么？默认给一个示例和一句提示
- **数据存不存、跑在哪、花不花钱**：前面已覆盖，确认都写进记录了

补默认时的语气：

> "有几个细节你刚才没提，但做出来必须得有，我先替你定了个常见的做法：
> · 处理错了能手动改 →（默认设成'能改'）
> · 表格列 →（默认'预置姓名/电话/数量，你能自己加'）
> 这样行吗？还是哪个想调？"

⚠️ **补默认、做简化时，绝不能动到核心能力。** 默认值只用来填周边的小空缺，不能把用户要验证的那件核心的事悄悄改掉。

将补充的默认项写入 `decisions.gaps_filled`。

## 五、对齐到一致 · 生成前的最后一道关

**没拆解清楚、没对齐到一致，绝不动手进入下一阶段。** 在结束阶段二之前，把整个需求的理解——**它是做什么的、端到端流程、核心能力、关键规则、技术方案、砍了什么留了什么**——用大白话**完整复述一遍**给 ta，请 ta 逐点确认：

> "动手生成文档之前，我把咱们理清的东西整个跟你过一遍：
> 1. **做什么**：______
> 2. **谁用**：______
> 3. **端到端流程**：______ → ______ → ______
> 4. **关键规则**：______
> 5. **做成什么**：______
> 6. **跑在哪**：______
> 7. **砍了什么**（这次不做）：______
>
> 这些都对吗？哪点不对、或我理解偏了，现在告诉我，我改。"

ta 指出哪里不对，就改，更新状态表，然后**再复述、再确认**——**反复，直到 ta 说"对，就是这样"为止**。
